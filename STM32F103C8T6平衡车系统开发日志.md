// 关键点2：自定义可变参数格式化输出函数 (摘取自USART1_Printf)
void USART1_Printf(char *format, ...)
{
	char String[128];                // 临时缓冲区
	va_list arg;                    // 定义可变参数列表指针
	va_start(arg, format);          // 初始化arg，指向format后的第一个参数
	vsprintf(String, format, arg);  // 将格式化字符串写入缓冲区
	va_end(arg);                    // 结束可变参数获取

### **遇到的问题与解决方案**：

**技术难点：实现灵活的格式化输出** 

**问题**：需要实现一个类似 `printf`的函数，以支持方便地输出调试信息（如：`“Pitch: %.2f\n”`），但又不希望直接使用体积庞大的标准库 `printf`。 

**解决方案**：如关键代码所示，采用 **可变参数函数** 实现。使用 `va_list`、`va_start`、`vsprintf`和 `va_end`这一套标准流程，将格式解析和变量参数处理在用户自定义函数内完成，最终仅发送格式化完毕的字符串，实现了功能与代码体积的平衡。

**硬件连接：CH340通信失败**

**问题**：在独立供电（平衡车用电池，CH340模块接电脑USB）的情况下，仅连接TX、RX，串口无法通信，上位机接收不到数据。 

**分析与解决**：在电气上，两个独立的系统要保证通信，必须要有共同的参考地（GND）。仅连接信号线而未连接地线，会导致双方电平判断标准不一致，从而通信失败。 

**解决方案**：将CH340模块的GND引脚与STM32系统的GND（如开发板的GND引脚）用杜邦线连接，建立共地。连接后通信立即恢复正常。

### 2026-04-12 | 非阻塞式按键驱动开发与状态机实现

### **今日核心目标**：

完成四路独立按键的非阻塞式检测驱动开发，实现按键的单击与长按（>1秒）识别，为系统提供可靠的人机输入接口，并为后续的菜单控制、参数调整等功能奠定基础。

### **具体工作内容**：

**硬件初始化**： 根据`Key.h`中的定义，将`Key1`(PB1)， `Key2`(PB0)， `Key3`(PA5)， `Key4`(PA4) 初始化为上拉输入模式。当按键未按下时，引脚被内部上拉电阻拉至高电平；按键按下时，引脚被拉至低电平。

**状态扫描函数 (`Key_Scan`)**： 该函数被放置在`TIM1`的1ms中断服务程序中，**每20ms调用一次**，实现了周期性的非阻塞扫描。 函数核心是一个**状态机**：通过对比本次读取的引脚状态(`StateX`)与上次保存的状态(`last_StateX`)，来检测按键的“按下边沿”和“释放边沿”，并分别设置`PressedX`和`ReleasedX`标志位。 在按键持续按下的周期中，软件计数器`countX`会递增，用于计时。

**模式判断函数 (`Key_Mode`)**： 此函数在主循环中被周期性调用。 其逻辑是：当检测到某个按键的`ReleasedX`（释放）标志被置位时，根据该按键按下过程中累计的`countX`值进行判断。 **判定逻辑**：若`countX >= 50`（即20ms * 50 = 1000ms），则判定为长按，`KeyX_Mode`赋值为`2`；否则判定为短按，赋值为`1`。判断完成后，将所有相关标志和计数器清零，为下一次按键事件做准备。

### **关键代码**：

```c
// 关键点1：Key_Scan函数中的状态机与边沿检测 (以Key1为例)
/* 获取按键上次状态 */
last_State1 = State1;
/* 读取当前实际引脚电平 */
State1 = GPIO_ReadInputDataBit(GPIOB, GPIO_Pin_1);

if(State1 == 0 && last_State1 == 1)  // 条件1：当前按下 & 上次松开 -> 检测到按下边沿
{
    Pressed1 = 1; // 设置“按下”事件标志
}
else if(State1 == 1 && last_State1 == 0) // 条件2：当前松开 & 上次按下 -> 检测到释放边沿
{
    Released1 = 1; // 设置“释放”事件标志
}
else if(State1 == 0 && last_State1 == 0) // 条件3：持续按下状态
{
    count1++; // 按下时间计数器累加 (每20ms加1)
}
// 注意：此设计有效消除了按键抖动的影响。

// 关键点2：Key_Mode函数中的长按/短按判决 (以Key1为例)
void Key_Mode(void)
{
    if(Released1) // 仅在按键释放后进行最终判决
    {
        if(!Key1_Mode && count1 >= 50)  // 长按判决
        {
            Key1_Mode = 2; // 长按状态码
            count1    = 0; // 计数器清零
            Pressed1  = 0; // 事件标志清零
            Released1 = 0;
        }        
        else if(!Key1_Mode && count1 < 50)  // 短按判决
        {
            Key1_Mode = 1; // 短按状态码
            count1    = 0;
            Pressed1  = 0;
            Released1 = 0;
        }       
    }
}
```

### **遇到的问题与解决方案**：

**问题**：在初步测试时，将`Key1_Mode`的数值实时显示在OLED上。发现长按超过1秒后，OLED能正确显示`2`（长按有效），但松开按键的瞬间，显示立刻变成了`1`（短按），逻辑出现错误。

**分析与调试**：此现象表明，在长按判决之后、按键松开之前，程序又进行了一次短按判决。检查原始代码逻辑发现，**长按判决和短按判决被放置在了不同的条件分支或执行时序中**。

**解决方案**：**将长按和短按的最终判决逻辑统一放置在“按键释放”(`ReleasedX == 1`)这个唯一的事件之后进行**，并保证只执行一次。如关键代码所示，在`Key_Mode`函数中，当检测到释放事件时，根据`countX`的值进行一次性的、排他性的判断（`countX >= 50`为长按，否则为短按），判决后立即将所有相关状态清零。这确保了一次按键动作只产生一个明确的状态结果（1或2）。

### **关键代码**：

```c
// 关键点1：I2C多字节读取函数的核心流程
uint8_t I2C_ReadBytes(uint8_t DevAddr, uint8_t RegAddr, uint8_t *Data, uint8_t Len) {
    // ... 发送起始条件、设备地址（写）、寄存器地址 ...
    I2C_GenerateSTART(I2C2, ENABLE); // 发送重复起始条件
    // ... 发送设备地址（读）...
    
    /* 关键逻辑：读取前 N-1 个字节，需回复ACK */
    if(Len > 1) {
        I2C_AcknowledgeConfig(I2C2, ENABLE);
        for(uint8_t i = 0; i < Len - 1; i++) {
            while(I2C_CheckEvent(I2C2, I2C_EVENT_MASTER_BYTE_RECEIVED) == ERROR);
            Data[i] = I2C_ReceiveData(I2C2);
        }
    }
    /* 关键逻辑：读取最后一个字节前，发送NACK和STOP信号 */
    I2C_AcknowledgeConfig(I2C2, DISABLE);
    I2C_GenerateSTOP(I2C2, ENABLE);
    while(I2C_CheckEvent(I2C2, I2C_EVENT_MASTER_BYTE_RECEIVED) == ERROR);
    Data[Len-1] = I2C_ReceiveData(I2C2);
    
    I2C_AcknowledgeConfig(I2C2, ENABLE); // 恢复ACK使能，为下次通信做准备
    return 0;
}

// 关键点2：从原始字节数据合成16位传感器数据
pData->Accel_X = (buffer[0] << 8) | buffer[1]; // 高字节在前，低字节在后
pData->Accel_Y = (buffer[2] << 8) | buffer[3];
pData->Accel_Z = (buffer[4] << 8) | buffer[5];
pData->Temp    = (buffer[6] << 8) | buffer[7];
pData->Gyro_X  = (buffer[8] << 8) | buffer[9];
pData->Gyro_Y  = (buffer[10] << 8) | buffer[11];
pData->Gyro_Z  = (buffer[12] << 8) | buffer[13];
```

### **具体工作内容**：

**PWM波形生成（`PWM.c/.h`）**： 

**定时器配置**：初始化`TIM2`，时钟源选择内部时钟。通过配置预分频器(PSC)=0和自动重装载寄存器(ARR)=7199，将PWM波形频率设置为10kHz。计算过程：定时器时钟为72MHz，PWM频率 = 72MHz / ((PSC+1)*(ARR+1)) = 72,000,000 / (1 * 7200) = 10,000 Hz。此频率可有效避免电机产生可闻噪音。

 **PWM通道配置**：配置`TIM2`的通道1(PA0)和通道2(PA1)为PWM模式1，输出极性高有效，并启用输出。通过`TIM_SetComparex()`函数动态修改比较寄存器(CCRx)的值，实现占空比0-100%的调节。

**电机驱动模块（`Motor.c/.h`）**：

 **硬件接口**：使用`GPIOB`的`Pin12~15`作为TB6612电机驱动芯片的控制引脚，分别控制左右电机的方向（AIN1/AIN2, BIN1/BIN2）。 

**功能抽象**：提供了高度封装的API函数。 Motor_Init(): 初始化PWM及控制GPIO。 Motor_Direction(Motor_Id, Direction): 控制电机转向（正转`FWD`、反转`REV`、制动`STOP`）。

**关键点**：代码中根据电机ID（`Motor_L`/`Motor_R`）分别映射到正确的GPIO，并考虑了左右电机安装方向相反导致的“正转”定义不同。 `Motor_Speed(Motor_Id, Duty)`: 设置指定电机的PWM占空比（0-100），实现调速。 `Motor_Stop()`: 同时制动两个电机（将驱动芯片输入置为同电平，电机快速停止）。

**系统集成与测试（`main.c`）**： 

在主函数中编写了测试循环，交替控制左右电机以20%的占空比进行正转、反转、停止各0.5秒的周期性运动，验证了驱动电路与代码逻辑的正确性。

### **今日核心目标**：

完成平衡车最核心的姿态感知环节。具体包括：优化底层驱动代码的稳定性与响应逻辑；基于MPU6050的原始数据，分别通过加速度计和陀螺仪计算俯仰角，并运用互补滤波算法进行数据融合，最终输出一个稳定、准确且动态响应良好的俯仰角，为后续的直立控制提供可靠的状态输入。



### **具体工作内容**：

**驱动层稳定性与逻辑优化**： 

**按键逻辑优化**：将`Key_Mode`函数的判断逻辑从“仅在松手(`Released`)时判决”优化为“长按超过1秒（`count >= 50`）时立即更新状态，短按仍在松手时判决”。此举使长按触发更及时，提升了人机交互体验。 **I2C通信加固**：在`MPU6050.c`的`I2C_ReadByte`和`I2C_ReadBytes`等函数中，将所有`while(I2C_CheckEvent(...))`循环替换为自定义的`WaitEvent`函数。该函数内置超时退出机制，有效防止了因I2C总线被意外干扰而导致的程序永久卡死，显著提升了系统鲁棒性。

**姿态解算算法实现与调试**：

 **传感器数据处理**： 加速度计俯仰角 (`AngleAcc`)：利用 `atan2(Accel_X, Accel_Z) * 180 / π`公式计算。`atan2`函数可正确处理四个象限，得到范围在-180°~180°之间的角度。此角度在静态时准确，但动态响应慢且易受线性加速度干扰。 陀螺仪俯仰角 (`AngleGyro`)：对陀螺仪Y轴（俯仰轴）的角速度进行积分。公式为：`AngleGyro = Angle + (-Gyro_Y / 32768 * 2000 * 0.01)`。其中，`-Gyro_Y`用于统一方向，`2000`对应量程±2000°/s，`0.01`是10ms的积分周期（`dt`）。此角度动态响应好，但存在积分漂移。 **互补滤波融合**：采用一阶互补滤波公式 `Angle = Kp * AngleAcc + (1 - Kp) * AngleGyro`。设置较小的`Kp=0.01`，意味着最终角度`Angle`主要由陀螺仪积分结果主导，同时用加速度计角度对其低频漂移进行微量修正，实现了优势互补。 **传感器校准**：在`MPU6050_GetRawData`函数中，对陀螺仪Y轴数据进行了零偏校准（`pData->Gyro_Y -= 43`），以抑制其零点漂移，这是减小姿态角稳态误差的关键步骤。

**系统调度与架构优化**：

 将`MPU6050_GetRawData`函数的调用从主循环移至`TIM1`的1ms定时中断服务函数中，并设置为每10ms执行一次。这为I2C总线操作提供了稳定的、可控的时序间隔，避免了主循环中高频调用可能导致的I2C状态机冲突，从系统架构上杜绝了卡死的风险。

**问题**：初期在主循环中无延迟地调用`MPU6050_GetRawData`，程序运行不久后卡死。 

**原因**：I2C通信是阻塞式且有时序要求的。在主循环中超高频率调用，会导致I2C状态机尚未完成上一次操作，就开始了下一次，标志位混乱，最终卡在等待事件的`while`循环中。

**串口打印触发硬错误（HardFault）**： 

**问题**：调用`USART1_Printf`输出角度值时，系统进入硬错误中断，程序崩溃。 

**原因**：`USART1_Printf`函数内部使用`vsprintf`将格式字符串写入一个固定大小的栈数组`String[128]`。当格式化输出的字符串长度超过128字节时，会发生栈缓冲区溢出，破坏相邻内存，从而触发硬错误。 

**解决方案**：将`vsprintf`替换为更安全的`vsnprintf`。`vsnprintf`的第三个参数指定了目标缓冲区的大小，可以防止写入越界。

void PID_Update(PID_struct *p)
{
    p->Error1 = p->Error0;                     // 保存上次误差
    p->Error0 = p->Target - p->Actual;         // 计算本次误差
    p->ErrorInt += p->Error0;                  // 误差积分（已做积分分离判断）

    p->Out = p->Kp * p->Error0

**电机控制参数异常**：

 **问题**：优化`Motor_Direction`函数后，传入的`Duty`参数值异常，与预期相差很大。 

**原因与解决**：检查发现，函数原型中的`Duty`参数类型仍为`uint8_t`。而PID输出为有符号整数，导致赋值时发生数据截断与符号解释错误。将形参类型修改为`int16_t`后，问题解决。

**电机运行中偶发卡顿/急停**：

 **问题**：调参过程中，电机在转动时会突然停止，造成车身剧烈抖动。 

**分析与调试**：在`PID_Angle_Update`函数中，用于暂存左右电机PWM计算结果的局部变量`PWM_L`, `PWM_R`, `AvePWM`, `DifPWM`在声明时未初始化。其初始值为栈内存中的随机数，在某些情况下会干扰最终计算结果，导致`Motor_Direction`函数收到接近0的占空比，触发停止逻辑。

 **解决**：在声明这些局部变量时，全部初始化为`0`。

**平衡车单向加速运动**：

 **问题**：调参时，小车总是缓慢地朝一个方向加速运动直至倒下，无法在零点附近维持。

 **原因**：当倾角很小时（如2°以内），PID计算出的`Duty`值很小（1~5）。此占空比低于电机的静摩擦力矩对应的启动电压，电机实际不转或转速极低，无法产生足够的回复力。

 **解决**：在`Motor_Direction`函数中增加死区与最小输出处理。将`1~5`和`-5~-1`区间的占空比统一提升至`5`和`-5`，并设置`-1~1`为死区。这确保了电机一旦启动即有足够的扭矩，同时消除了零点附近的高频开关抖动。

**平衡点存在稳态偏差**： 

**问题**：即使调整PID参数，平衡车稳定的角度点并非竖直的0度，而是有一个固定的偏差。

 **原因**：MPU6050传感器存在安装误差，或加速度计零点标定不精确，导致`AngleAcc`计算出的角度在物理竖直时不为0。这个偏差会通过互补滤波传递给`Angle`，使得控制器认为“平衡状态”是一个有倾角的状态。 

**解决**：在`MPU6050.c`的`MPU6050_GetRawData`函数中，对计算出的`AngleAcc`进行软件调零：`AngleAcc -= 0.5;`。通过实验测得竖直状态下的角度值，并用此偏移量进行补偿，使传感器读数与物理状态对齐。

if (NRF24L01_Receive() == 1) // 成功接收到一包数据
{
    uint8_t ID = NRF24L01_RxPacket[0]; // 协议：第一个字节为ID
    if (ID == 0x00) // 验证ID，防止干扰
    {
        int8_t LH = NRF24L01_RxPacket[1]; // 协议：第二个字节为左右摇杆水平量 (控制速度)
        int8_t RV = NRF24L01_RxPacket[4]; // 协议：第五个字节为右摇杆垂直量 (控制转向)
        
        PID_Speed.Target = LH;       // 遥控器直接设定速度目标值
        PID_Turn.Target  = -(RV * 10); // 遥控器设定转向目标值（比例放大）
    }            
}

// 直立环中集成转向环输出，实现差速控制
void PID_Angle_Update(void)
{
    // ... 直立环PID计算，得到 AvePWM ...
    int16_t AvePWM = -PID_Angle.Out; // 基础PWM，用于维持平衡
    int16_t DifPWM = PID_Turn.Out;   // 转向环输出的差速量
    
    // 核心：将转向差速量叠加到左右电机基础PWM上
    PWM_L = AvePWM + DifPWM / 2; // 左电机PWM
    PWM_R = AvePWM - DifPWM / 2; // 右电机PWM
    // ... 限幅后驱动电机 ...
}
